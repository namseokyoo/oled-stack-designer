# OLED 구조 생성기 에이전트화 설계안

작성 기준: `docs/PRD.md`의 FR-010(자연어 입력) 및 v2.0 후속 요구사항을 기반으로, 현재 제품 상태를 v1.6.0 완료로 가정하고 작성함.

---

## 1. 배경 및 목표

현재 OLED Stack Designer는 v1.6.0 기준으로 다음 강점을 이미 확보한 상태다.
- 구조적 데이터 모델 기반 편집
- Single / RGB 구조 모드
- 레이어 CRUD, FMM 전환, Undo/Redo
- PNG/SVG 내보내기와 JSON 저장/불러오기
- 하단 자연어 입력 바를 수용할 수 있는 UI 방향성(PRD 4.1)

반면 FR-010은 아직 "자연어로 레이어 추가/수정/삭제" 수준으로만 정의되어 있어, 실제 사용자 기대인 "구조를 설명하면 적절한 OLED 스택을 제안·생성해주는 기능"과는 간극이 있다.

따라서 다음 단계의 제품 요구사항은 단순한 자연어 명령 실행기가 아니라, 아래 두 역할을 수행하는 "구조 생성기 에이전트"를 정의해야 한다.
1. 사용자의 자연어 의도를 구조 편집 액션으로 변환하는 명령형 에이전트
2. 목표 성능/색상/구조 힌트를 바탕으로 초기 스택 초안을 생성하는 생성형 에이전트

핵심 제품 목표는 다음과 같다.
- 사용자가 빈 캔버스에서 시작하지 않아도 되게 한다.
- OLED 비전문 UI 조작 대신, 연구자가 익숙한 서술형 언어로 구조를 만든다.
- 생성 결과를 곧바로 기존 편집기/내보내기/Undo 시스템과 연결한다.
- 시뮬레이션 도구로 범위가 확장되지 않도록 "편집 보조 에이전트"로 포지셔닝한다.

## 2. FR-010 및 v2 요구사항과의 연결 해석

### FR-010의 현재 의미
PRD의 FR-010은 다음 4가지를 최소 조건으로 둔다.
- 하단 입력 바에서 자연어 명령 입력
- 레이어 추가/수정/삭제 예시 지원
- LLM API 연동
- 실패 시 에러 피드백

이는 본질적으로 "한 줄 명령 해석기"에 가깝다.

### 에이전트화 시 확장 해석
에이전트화는 FR-010을 아래 범위까지 확장해야 한다.
- 단일 명령 해석 → 다단계 구조 생성 워크플로
- 레이어 단건 수정 → 구조 초안 생성 + 사용자 확인 + 적용
- 실패 피드백 → 모호성 질의응답 + 안전한 수정 제안
- 단순 LLM 호출 → 도메인 제약을 반영한 구조화된 액션 플래너

### v2 항목과의 연결
PRD v2.0 장기 항목 중 구조 생성기 에이전트화와 직접 연결되는 것은 다음과 같다.
- 자연어 입력 (LLM 연동): 핵심 선행 요구사항
- 프리셋 템플릿 라이브러리: 에이전트의 생성 결과를 보정/초기화하는 지식 소스
- 구조 diff 비교 상세: 에이전트 제안 전/후 비교 및 승인 UX의 기반
- Presentation Mode: 직접 선행은 아니지만, 에이전트가 생성한 구조를 전달력 있게 표현하는 후속 가치

즉, 구조 생성기 에이전트는 v2 기능들 사이에서 독립 기능이 아니라, 템플릿/비교/자연어 입력을 묶는 상위 사용자 경험으로 설계하는 것이 적절하다.

## 3. 사용자 시나리오

### 시나리오 A. 빈 캔버스에서 기본 OLED 구조 초안 생성
- 사용자: OLED 연구자
- 상황: 논문 Figure 초안을 빨리 만들어야 함
- 입력: "기본 bottom-emission OLED 구조 만들어줘. ITO / HIL / HTL / EML / ETL / EIL / Al 순서로"
- 기대 결과:
  - 에이전트가 Single Stack 초안을 생성
  - 각 레이어 이름/role 기본값 자동 매핑
  - 두께/재료가 빠진 항목은 빈 값 또는 추천값으로 표시
  - 사용자에게 "초안 생성됨"과 함께 적용 전 미리보기 제공
- 핵심 가치: 빈 캔버스 진입 장벽 제거

### 시나리오 B. 재료와 두께를 포함한 구체 구조 생성
- 사용자: 디스플레이 엔지니어
- 입력: "ITO 150nm, HAT-CN 5nm, NPB 40nm, Ir(ppy)3 도핑 EML 30nm, BPhen 35nm, LiF 1nm, Al 100nm 구조 생성"
- 기대 결과:
  - 순서/두께/재료명 파싱
  - 적절한 role 추론(예: NPB→HTL, BPhen→ETL)
  - 불명확 항목은 custom role 또는 확인 질문으로 처리
  - 결과를 기존 Layer 데이터 모델로 변환해 편집 가능 상태로 반영
- 핵심 가치: 텍스트 사양서를 바로 시각 구조로 전환

### 시나리오 C. RGB FMM 구조 생성
- 사용자: 연구원 또는 장비/소재 업체
- 입력: "RGB FMM 구조 생성해줘. 공통 HTL 위에 Red/Green/Blue EML 각각 25/20/22nm로"
- 기대 결과:
  - structureMode를 rgb로 전환
  - 공통층과 채널별 개별층(appliesTo, channelOverrides) 구성
  - 공통층/개별층 규칙은 기존 데이터 모델에 맞춰 생성
  - 생성 후 공통층 통합 선택과 FMM 전환이 기존 UX 그대로 동작
- 핵심 가치: 복잡한 RGB 구조를 수동 편집 없이 시작 가능

### 시나리오 D. 기존 구조 수정 에이전트
- 입력: "HTL을 50nm로 바꾸고 EML 재료를 CBP로 수정해줘"
- 기대 결과:
  - 해당 레이어 식별
  - 수정 계획을 2개 액션으로 표시
  - 적용 후 Undo/Redo 히스토리에 한 묶음 또는 다중 액션으로 기록
- 핵심 가치: FR-010의 본래 목적을 충족하면서도 안전한 편집 보조 제공

### 시나리오 E. 모호한 요청에 대한 질의응답
- 입력: "green device 구조 만들어줘"
- 기대 결과:
  - 에이전트가 즉시 추정 적용하지 않고 질문
  - 예: "Single 구조로 생성할까요? 기본 7층 OLED를 기준으로 Green EML만 지정할까요?"
  - 사용자가 답하면 초안 재생성
- 핵심 가치: 잘못된 구조 자동 생성 방지

### 시나리오 F. 템플릿 기반 생성
- 입력: "논문용 기본 OLED 템플릿으로 시작하고 HTL만 TAPC로 바꿔줘"
- 기대 결과:
  - 예제/프리셋 라이브러리를 검색 기반 컨텍스트로 활용
  - 템플릿 복제 후 수정안 적용
- 핵심 가치: 프리셋 템플릿 라이브러리와 에이전트의 결합

### 시나리오 G. 변경 비교 후 승인
- 입력: "이 구조를 RGB 구조로 변환해줘"
- 기대 결과:
  - 에이전트가 직접 반영하지 않고 before/after diff 제공
  - 변경 레이어 수, 공통층 분리 여부, 채널별 override를 설명
  - 사용자가 승인해야 실제 적용
- 핵심 가치: 구조 diff 비교 상세(v2)와 직접 연결

## 4. 에이전트화 MVP 범위

### 4.1 MVP에서 반드시 포함할 범위

#### A. 입력 범위
- 단일 턴 자연어 입력 1개
- 한국어/영어 혼합 명령 허용
- 입력 유형은 아래 3종으로 제한
  1. 새 구조 생성
  2. 기존 구조 수정
  3. 구조 설명/요약 요청

#### B. 지원 구조 범위
- Single Stack 생성
- RGB FMM 기본 구조 생성
- 레이어 역할(role): cathode/eil/etl/eml/htl/hil/anode/custom
- 시뮬레이션, 전기적 성능 예측, 공정 최적화는 제외

#### C. 에이전트 출력 범위
에이전트는 자유 텍스트만 반환하면 안 되고 아래 중 하나를 반환해야 한다.
- `create_structure` 제안
- `edit_layers` 제안
- `ask_clarification` 제안
- `reject_request` 제안

즉 제품 관점의 핵심은 "채팅 응답"이 아니라 "구조화된 실행 의도"다.

#### D. 사용자 승인 흐름
- 생성/수정은 기본적으로 "초안 제안 → 사용자 확인 → 적용" 2단계로 동작
- 단순 명령(예: "HTL 50nm로")만 선택적으로 즉시 적용 가능
- 파괴적 변경(삭제, RGB 변환, 다중 레이어 수정)은 항상 확인 필요

#### E. 도메인 지식 소스
MVP에서는 아래 2개만 사용한다.
- 현재 프로젝트의 실제 구조 데이터(JSON 상태)
- 앱 내 role/템플릿/예제 기반 사전 정의 규칙

외부 논문 검색, 웹 검색, 재료 DB 실시간 질의는 제외한다.

### 4.2 MVP에서 제외할 범위
- 성능/효율/수명에 대한 과학적 추천 엔진
- "최적 구조 추천"처럼 검증 불가능한 생성
- 다중 턴 장기 메모리
- 음성 입력
- 클라우드 협업
- 외부 문헌 인용 기반 자동 구조 생성
- Compare 모드 전체 자동 해석
- PDF/CSV 생성 명령까지 포함한 범용 에이전트화

### 4.3 MVP 성공 기준
- 사용자가 30초 내 빈 캔버스에서 기본 구조 초안을 생성할 수 있다.
- 자연어 입력으로 단일/복합 수정이 Undo 가능한 액션으로 적용된다.
- 모호한 요청에 대해 잘못된 구조를 임의 생성하지 않는다.
- 에이전트 생성 결과가 기존 저장/불러오기/내보내기와 충돌하지 않는다.

## 5. 제품 설계 포인트

### 5.1 사용자 경험 설계 원칙
1. 에이전트는 편집기를 대체하지 않고 가속한다.
2. 사용자는 항상 최종 적용권을 가진다.
3. 에이전트가 추론한 값과 사용자가 명시한 값은 구분해 보여준다.
4. 생성된 모든 결과는 기존 레이어 모델과 완전 호환되어야 한다.
5. 잘 모르겠는 경우 생성보다 질문을 우선한다.

### 5.2 추천 인터랙션 모델
하단 입력 바를 아래 3상태로 확장하는 것이 적합하다.
- Idle: 예시 프롬프트 제안
- Draft Review: 에이전트가 만든 구조/변경안 미리보기
- Clarification: 부족한 정보 질문

### 5.3 추천 정보 아키텍처
에이전트 응답은 최소 아래 요소를 포함해야 한다.
- intent: create / edit / explain / clarify / reject
- target mode: single / rgb / current
- structured actions: 생성/수정 액션 목록
- confidence: high / medium / low
- assumptions: 추론한 값 목록
- warnings: 불확실하거나 위험한 변경

### 5.4 왜 "명령 해석기"가 아니라 "에이전트"인가
단순 명령 해석기는 사용자가 모든 구조 세부사항을 직접 써야 한다.
반면 구조 생성기 에이전트는 아래 가치를 제공한다.
- 누락 필드 보완
- OLED 도메인 role 추론
- RGB/FMM 데이터 모델 자동 구성
- 변경 diff 및 확인 흐름 제공
- 템플릿과 사용자 의도 결합

## 6. 비기능 요구사항

### NFR-A1. 구조 안전성
- 에이전트가 적용하는 모든 변경은 현재 JSON 데이터 모델 스키마를 만족해야 한다.
- 유효하지 않은 role, 음수 두께, 지원하지 않는 appliesTo 조합은 적용 전에 차단해야 한다.
- 실패 시 부분 적용 없이 트랜잭션 단위로 롤백해야 한다.

### NFR-A2. 설명 가능성
- 사용자가 "왜 이렇게 생성됐는지" 이해할 수 있도록 가정값과 추론 근거를 표시해야 한다.
- 재료명으로 role을 추정한 경우 명시적으로 표시해야 한다.

### NFR-A3. 성능
- 일반적인 단일 요청의 응답 시작 시간은 3초 이내, 적용 완료까지 5초 이내를 목표로 한다.
- LLM 응답 지연 시 로딩 상태와 취소 옵션을 제공해야 한다.
- UI 편집 성능(NFR-001)을 저해하지 않도록 LLM 처리와 캔버스 렌더링을 분리해야 한다.

### NFR-A4. 신뢰성
- 동일 입력에 대해 지나치게 다른 구조가 생성되지 않도록 생성 자유도를 제한해야 한다.
- 템플릿/룰 기반 후처리로 출력 일관성을 보정해야 한다.
- 애매하거나 위험한 요청은 보수적으로 질의응답 흐름으로 전환해야 한다.

### NFR-A5. 데이터 무결성 및 히스토리
- 에이전트 적용 결과는 Undo/Redo에서 일반 편집과 동일하게 동작해야 한다.
- 생성/수정 전 상태를 반드시 히스토리에 저장해야 한다.
- 저장/불러오기 후에도 에이전트 생성 구조가 손실 없이 유지되어야 한다.

### NFR-A6. 프라이버시 및 배포 제약
- PRD 5.3의 "오프라인 동작 필수, LLM 기능 제외" 제약과 충돌하지 않도록 제품 옵션을 분리해야 한다.
- 즉, 기본 앱은 오프라인 편집 가능, 에이전트 기능은 온라인 옵션 기능으로 명확히 고지해야 한다.
- 조직/기업 사용자를 위해 향후 로컬 모델 또는 BYO API 키 구조를 고려해야 한다.

### NFR-A7. 감사 가능성
- 최소한 최근 요청/적용 결과/실패 사유를 로컬 로그로 남겨야 한다.
- 잘못된 생성 사례를 재현할 수 있어야 프롬프트/룰 튜닝이 가능하다.

### NFR-A8. 국제화 입력 대응
- 한국어/영어 혼합 재료명과 단위 표현(nm, 나노, thickness)을 허용해야 한다.
- 대소문자, 슬래시, 하이픈, 도핑 표기 등의 재료명 변형을 최대한 보존해야 한다.

### NFR-A9. 내보내기 정합성 유지
- 에이전트가 생성한 구조도 기존 geometry generator를 그대로 사용해야 한다.
- 화면에서 승인된 구조가 PNG/SVG 내보내기에서 동일하게 반영되어야 한다.

## 7. 단계별 로드맵

### Phase 0. 코드/로직 리팩토링 (선행 정비)
목표: 현재 편집기 로직을 에이전트가 안전하게 재사용할 수 있는 구조로 정리

## 8. 제품 의사결정 제안

### 제안 1. 기능명은 "자연어 입력"보다 "AI 구조 생성기"가 적합
사용자 기대 가치를 더 정확히 전달한다. 다만 내부 요구사항 번호는 FR-010의 확장판으로 유지 가능하다.

### 제안 2. 첫 출시에서는 "자동 적용"보다 "초안 제안"을 기본값으로 둔다
과학/엔지니어링 사용성에서는 빠름보다 신뢰가 우선이다.

### 제안 3. 에이전트 품질의 핵심은 모델 성능보다 출력 스키마와 룰 보정이다
OLED Stack Designer는 범용 채팅앱이 아니라 구조 편집기이므로, 자유 텍스트보다 구조화된 액션이 우선이다.

### 제안 4. 템플릿 라이브러리는 v2의 별도 기능이 아니라 에이전트 정확도를 높이는 핵심 자산으로 취급해야 한다
템플릿이 있어야 생성 결과가 일관되고, diff UX도 단순해진다.

### 제안 5. 성공 지표는 "얼마나 똑똑해 보이냐"가 아니라 "얼마나 빨리 올바른 초안을 만들게 하느냐"로 둔다
추천 KPI:
- 첫 구조 생성까지 걸린 시간
- 자연어 요청 1회당 수동 수정 횟수
- 승인 없이 폐기된 에이전트 제안 비율
- 잘못된 role 추론 비율

## 9. 최종 요약

구조 생성기 에이전트화는 단순한 FR-010 구현이 아니라, v1.6.0의 구조 편집기 위에 "초안 생성 + 안전한 수정 제안 + 확인 적용" 레이어를 얹는 작업이다.

MVP는 아래에 집중하는 것이 가장 적절하다.
- Single/RGB 기본 구조 초안 생성
- 기존 구조에 대한 자연어 수정
- 모호할 때 질문하는 안전한 에이전트
- diff/승인/Undo와 연결된 구조화된 액션 시스템

이 방향은 PRD의 기존 포지셔닝인 "stack-aware editor + editable export"와 일관되며, 시뮬레이션 제품으로 범위가 비대해지는 것을 막으면서도 v2 자연어 입력을 실질적 사용자 가치로 전환한다.
