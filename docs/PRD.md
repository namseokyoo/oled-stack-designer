## 1. 개요 (Overview)

### 1.1 프로젝트 목적

OLED 소자의 레이어 스택 구조를 **빠르고 직관적으로** 시각화·편집·내보내기할 수 있는 Windows 데스크탑 앱을 개발한다.

현재 OLED 연구자와 엔지니어들은 PowerPoint, Inkscape 등 범용 도구로 소자 구조도를 그리고 있다. 이 과정은 비효율적이며 구조적 데이터(두께, 재료명)와 시각 표현이 분리되어 있다. OLED Stack Designer는 **데이터 기반 스택 편집기 + 고품질 벡터 내보내기**로 이 문제를 해결한다.

**핵심 가치**: "PowerPoint에서 박스 그리는 걸, 구조적으로 빠르게"

**사용자용 가치 제안**: "소재명·두께·레이어 순서를 입력하면, 논문에 바로 쓸 수 있는 OLED 스택 다이어그램을 만들어줍니다."

### 1.2 타겟 사용자

| 사용자 | 주요 용도 |
|--------|----------|
| OLED 연구자 | 논문용 구조도, 실험 조건 비교 |
| 디스플레이 엔지니어 | 개발 보고서, 특허 도면, 사내 문서 |
| 장비/소재 업체 | 제안서, 마케팅 기술자료, 교육자료 |

### 1.3 성공 지표

| 지표 | 목표값 |
|------|--------|
| MVP 완성 | 14일 이내 |
| 핵심 기능 동작 | Single + RGB 2가지 뷰 (Compare는 v1.1) |
| 내보내기 품질 | 논문 게재 가능 수준 (300dpi+ 벡터) |

## 2. 기능 요구사항 (Functional Requirements)

### FR-001: 레이어 CRUD
- **설명**: 레이어 추가, 삭제, 이름/두께/재료명/색상 편집
- **우선순위**: 필수
- **인수 조건**:
  - [ ] 레이어 추가 시 이름, 두께(nm), 재료명, 색상 입력 가능
  - [ ] 레이어 삭제 시 확인 다이얼로그
  - [ ] 인라인 편집 (더블클릭) 지원
  - [ ] 레이어 복제: 우클릭 메뉴(FR-016) 또는 Ctrl+D로 선택 레이어 복제

### FR-002: 레이어 순서 변경 (드래그앤드롭)
- **설명**: 레이어 블록을 드래그하여 순서 변경
- **우선순위**: 필수
- **인수 조건**:
  - [ ] 마우스 드래그앤드롭으로 순서 변경
  - [ ] 키보드 대안 (Alt+↑/↓)
  - [ ] 드래그 시 시각적 피드백 (반투명 + 삽입 위치 표시)

### FR-003: 구조 모드 전환
- **설명**: Single Stack / RGB 통합 2가지 뷰 (MVP), Compare는 v1.1
- **우선순위**: 필수 (Single/RGB), 권장 (v1.1, Compare)
- **인수 조건**:
  - [ ] 상단 탭으로 모드 전환: **Single / RGB** (MVP)
  - [ ] **Single Stack**: 단일 스택 편집 (기본)
  - [ ] **RGB 통합**: 공통층 + R/G/B 개별층 (FMM 구조)
  - [ ] **Compare** (v1.1): 2~4개 소자 Side-by-side 비교

### FR-004: RGB 자연 블록 스택 렌더링
- **설명**: RGB 모드에서 EML 두께 차이에 따른 공통층 자연 적층 표현
- **우선순위**: 필수
- **인수 조건**:
  - [ ] EML 아래 공통층: 전체 가로폭 평평한 rect
  - [ ] EML: R/G/B 3열, 하단 정렬, 각각 두께에 비례한 높이
  - [ ] EML 위 공통층: R/G/B 3열, 각 EML 위에 빈틈 없이 올라앉음 (자연 블록 스택)
  - [ ] 같은 레이어 = 같은 색상 (보조 연결 장치 없음)
  - [ ] 공통층 클릭 시 3개 rect 동시 선택 + 속성 통합 관리
  - [ ] 공통층 라벨은 1개만 표시 (중앙 기준)

### FR-005: 두께 모드 전환
- **설명**: 균일 두께 모드 ↔ 실제 두께 비례 모드 전환
- **우선순위**: 필수
- **인수 조건**:
  - [ ] 상단 토글 스위치로 전환
  - [ ] 균일 모드: 모든 레이어 동일 높이
  - [ ] 실제 두께 모드: nm 비례 높이 (최소 height 보장)
  - [ ] 전환 시 애니메이션 (300ms ease)

### FR-006: 레이블링
- **설명**: 레이어별 이름, 두께(nm), 재료명 표시
- **우선순위**: 필수
- **인수 조건**:
  - [ ] 레이어 블록 내부: 레이어 이름
  - [ ] 블록 우측 또는 내부: 두께(nm) + 재료명
  - [ ] 두께가 얇을 때 레이블 자동 축소 또는 툴팁 전환

### FR-007: 내보내기
- **설명**: 다이어그램을 다양한 형식으로 내보내기
- **우선순위**: 필수
- **인수 조건**:
  - [ ] PNG: 캔버스 크기 × 2배율 (Retina 대응), 최소 2000px 폭
  - [ ] SVG: viewBox 정의, 폰트 인라인화, 스타일 인라인화 필수
  - [ ] SVG: Illustrator/Inkscape에서 편집 가능 확인
  - [ ] 화면 렌더러와 내보내기가 동일 geometry generator 공유 (불일치 테스트 필수)
  - [ ] 내보내기 파일에 모든 레이블 포함
  - [ ] 내보내기 배경색 선택: 흰 배경 (논문용) / 다크 배경 (발표용) / 투명 배경 (합성용)
  - [ ] 흰 배경 모드: 레이어 색상 유지, 텍스트·레이블 색상 자동 반전 (다크→라이트)

### FR-008: 저장/불러오기
- **설명**: 프로젝트 파일 저장 및 불러오기 (MVP: 수동 저장만)
- **우선순위**: 필수 (수동 저장), 권장 (v1.1, 자동 저장)
- **인수 조건**:
  - [ ] JSON 형식 프로젝트 파일 저장
  - [ ] 파일 불러오기 시 모든 레이어 정보 손실 없이 복원
  - [ ] "다른 이름으로 저장" (Save As) 지원 — Ctrl+Shift+S
  - [ ] 원본 파일 덮어쓰기 방지

### FR-008a: 자동 저장
- **설명**: 일정 간격 자동 저장
- **우선순위**: 권장 (v1.1)
- **인수 조건**:
  - [ ] 5분 간격 자동 저장
  - [ ] 크래시 시 데이터 손실 최소화

### FR-009: Undo/Redo
- **설명**: 작업 실행취소/재실행
- **우선순위**: 필수
- **인수 조건**:
  - [ ] Ctrl+Z (Undo), Ctrl+Y (Redo)
  - [ ] 최소 20단계 지원
  - [ ] 레이어 CRUD, 순서 변경, 속성 편집 모두 대상
  - [ ] 메모리 사용량: 100개 히스토리 항목 기준 50MB 이하
  - [ ] Undo 대상: 레이어 CRUD, 순서 변경, 속성 편집, 레이어 복제, FMM 전환, 구조 모드 전환(Single↔RGB)
  - [ ] Undo 비대상: 내보내기, 저장, 두께 모드 전환(시각적 표시 변경일 뿐 데이터 변경 아님)

### FR-010: 자연어 입력 (LLM 연동)
- **설명**: 자연어로 레이어 추가/수정/삭제
- **우선순위**: 권장 (MVP 후속)
- **인수 조건**:
  - [ ] 하단 입력 바에서 자연어 명령 입력
  - [ ] 예: "HTL 레이어 50nm NPB 재료로 추가", "EML을 Red에서 Green으로 변경"
  - [ ] LLM API 연동 (모델/API는 추후 결정)
  - [ ] 명령 실패 시 에러 피드백

### FR-011: 비교 모드 상세
- **설명**: 2~4개 소자 구조 Side-by-side 비교
- **우선순위**: 권장 (v1.1)
- **인수 조건**:
  - [ ] 소자별 DeviceCard 수평 배치
  - [ ] 차이점 시각 표시 (두께 다름, 재료 다름, 레이어 추가/삭제)
  - [ ] 우측 패널에 수치 비교 테이블

### FR-012: 추가 내보내기
- **설명**: PDF, CSV 내보내기
- **우선순위**: 선택 (v2)
- **인수 조건**:
  - [ ] PDF 내보내기 (논문/보고서 첨부용)
  - [ ] CSV 내보내기 (레이어 목록 + 두께 데이터 표 형식)

### FR-013: 총 두께 합계 표시
- **설명**: 스택 전체 두께 합계 실시간 표시
- **우선순위**: 필수
- **인수 조건**:
  - [ ] 우측 패널 또는 상태 바에 총 두께(nm) 표시
  - [ ] 레이어 추가/삭제/두께 변경 시 실시간 갱신

### FR-014: 예제 프로젝트 파일
- **설명**: 앱과 함께 예제 .json 파일 3개를 제공하여 빠른 시작 지원. 앱 내장 템플릿이 아닌, 불러오기(FR-008)로 활용하는 외부 파일 형태.
- **우선순위**: 필수
- **인수 조건**:
  - [ ] 예제 파일 3개 제공: (1) 기본 OLED 단일 스택 7층 (ITO/HIL/HTL/EML/ETL/EIL/Al), (2) RGB FMM 구조 예제, (3) 빈 최소 구조 (Anode/EML/Cathode 3층)
  - [ ] 앱 설치 디렉토리 또는 문서 폴더에 examples/ 폴더로 배치
  - [ ] 첫 실행 시 Empty State에서 "예제 파일 열기" 버튼 표시 → 파일 불러오기(FR-008) 활용

### FR-015: Empty State 안내
- **설명**: 레이어가 0개일 때 캔버스에 시작 안내 표시
- **우선순위**: 필수
- **인수 조건**:
  - [ ] 빈 캔버스 중앙에 안내 메시지: "레이어를 추가해서 OLED 스택을 만들어보세요"
  - [ ] "예제 파일 열기" 버튼 (FR-014 예제 파일로 연결)
  - [ ] "+ 새 레이어 추가" 버튼
  - [ ] 레이어 1개 이상 추가되면 안내 자동 숨김

### FR-017: 색상 팔레트 전환
- **설명**: 레이어 색상 팔레트를 3종 중 선택하여 전체 레이어 색상 일괄 변경
- **우선순위**: 필수
- **인수 조건**:
  - [ ] 3종 팔레트 제공:
    - Classic: OLED 업계 관례 색상 (진한 과학적 톤) — 기본값
    - Pastel: 부드러운 파스텔 톤 (발표/교육용)
    - Vivid: 최대 채도 네온 톤 (마케팅/포스터용)
  - [ ] 팔레트 전환 시 전체 레이어 색상 일괄 변경 (CSS Custom Properties 기반)
  - [ ] 각 팔레트별 10종 레이어 색상 (Cathode, EIL, ETL, EML-B/G/R, CGL, HTL, HIL, Anode)
  - [ ] 개별 레이어 색상 커스텀도 가능 (팔레트와 독립적 오버라이드)
  - [ ] 내보내기 시 현재 적용 팔레트 색상 그대로 반영

### FR-016: 레이어 컨텍스트 메뉴 (우클릭)
- **설명**: 레이어 블록에서 우클릭 시 컨텍스트 메뉴 표시. 복제, FMM 전환, 삭제 등 주요 조작을 통합 제공.
- **우선순위**: 필수
- **인수 조건**:
  - [ ] 우클릭 메뉴 항목:
    - 레이어 복제 (Duplicate) — Ctrl+D 단축키 병행
    - FMM 개별층으로 전환 (공통층 → R/G/B 개별) — RGB 모드에서만 표시
    - 공통층으로 전환 (개별층 → 공통) — RGB 모드에서만 표시
    - 레이어 잠금/잠금 해제
    - 레이어 삭제 — Delete 키 단축키 병행
  - [ ] Single 모드에서는 FMM 관련 메뉴 항목 숨김
  - [ ] 공통층(appliesTo: ['r','g','b'])인 경우 "FMM 개별층으로 전환" 표시
  - [ ] 개별층(appliesTo: ['r'] 등)인 경우 "공통층으로 전환" 표시
  - [ ] FMM 전환 시 확인 다이얼로그 (데이터 변환 안내)
  - [ ] Undo/Redo 대상에 포함

## 3. 비기능 요구사항 (Non-Functional Requirements)

### NFR-001: 성능
- 레이어 50개 환경에서 드래그앤드롭 60fps 유지
- 레이어 100개에서 렌더링 3초 이내
- 저장/불러오기 2초 이내
- backdrop-filter 적용 요소: 동시 3개 이하 (패널 + 툴바 + 메뉴)
- 레이어 블록에 backdrop-filter 적용 금지 (N개 동시 렌더링 시 성능 저하)
- prefers-reduced-motion 미디어 쿼리 존중: 블러/애니메이션 비활성화 옵션

### NFR-002: 호환성
- Windows 10/11 지원
- 최소 해상도: 1024×768
- 권장 해상도: 1280×800 이상
- DPI 150%, 200% 스케일링 대응

### NFR-003: 데이터 무결성
- 저장 → 불러오기 시 모든 레이어 정보 100% 복원
- MVP에서는 수동 저장 방식. 저장하지 않고 종료 시 확인 다이얼로그 표시. 자동 저장은 v1.1에서 도입 예정
- 편집 중 5분 간격으로 임시 백업 파일 1건 자동 생성 (앱 임시 디렉토리)
- 앱 비정상 종료 후 재실행 시 "복구 가능한 백업이 있습니다" 안내
- 자동 저장(v1.1)과 별개의 크래시 복구 전용 기능

### NFR-004: 내보내기 정합성
- **원칙**: 내보내기는 과학적 정확성 우선
- 레이어 색상, 위치, 크기, 레이블: 화면과 내보내기 100% 일치
- 레이어 블록 상단 하이라이트 그라디언트: 화면 전용 장식 → 내보내기 시 단색 폴백 (과학 문서 호환)
- UI 크롬(글라스모피즘, 그림자, 블러): 내보내기에 포함하지 않음
- geometry generator는 화면/내보내기 공유 (위치·크기 불일치 방지)

## 4. UI/UX 요구사항

### 4.1 레이아웃
- **2-패널**: 중앙 캔버스 + 우측 속성 패널 (260px)
- 상단 최소 툴바: 구조 탭 (Single / RGB) + 두께 토글 + 내보내기
- 하단 자연어 입력 바 (v2에서 LLM 연동)
- 좌측 패널 **없음** (심플한 UI)

### 4.2 디자인 톤
- **전체 톤**: 세련되고 현대적 — Apple visionOS 글라스모피즘 참조
- **배경**: 다크 테마 (`oklch(14% 0.01 250)`)
- **참조 디자인**: Apple visionOS, Linear, Vercel Dashboard + ChemDraw (과학적 정밀성)
- **톤앤매너**: Precision(정밀), Elegant(우아), Depth(깊이감), Scientific(과학적)
- **핵심 원칙**: "캔버스(레이어)는 정확, UI 크롬은 세련"
  - 캔버스: 단색 레이어 블록 + 정밀한 데이터 표현
  - UI 프레임(패널/메뉴/툴바): 글라스모피즘(반투명 + 블러 + 깊이감)

#### 글라스모피즘 적용 범위
| 요소 | MVP 적용 | 후순위(v1.1) | backdrop-filter |
|------|----------|-------------|----------------|
| 우측 속성 패널 | ✅ | - | blur(24px), 75% 불투명 |
| 상단 툴바 | ✅ | - | blur(16px), 85% 불투명 |
| 컨텍스트 메뉴 | - | ✅ | blur(32px), 80% 불투명, radius 12px |
| 선택 레이어 글로우 | - | ✅ | 파란 글로우 그림자 |
| 레이어 블록 (캔버스) | ❌ | ❌ | 단색 유지 (SVG 내보내기 호환 + 성능) |

#### 모서리/그림자 체계
- border-radius: 6px(작은 요소) ~ 20px(메뉴/모달), visionOS 스타일 둥근 모서리
- 그림자 4단계: sm / md / lg / glow (선택 레이어 파란 글로우)
- 선택 레이어 글로우: OLED 발광을 은유적으로 표현

#### 애니메이션
- 레이어 삽입/삭제: 스프링 이징 ("쏙" 하는 탄성 느낌)
- 패널/모드 전환: 부드러운 ease (220ms)
- 두께 모드 전환: 380ms ease

#### 성능 대응
- 저사양 PC 대비 블러 비활성화 옵션 (prefers-reduced-motion 존중)
- backdrop-filter는 고정 UI 요소에만 적용 (레이어 블록 적용 금지)

### 4.3 핵심 인터랙션
- 레이어 추가: 캔버스 내 "+" 버튼
- 레이어 이동: 드래그앤드롭 또는 Alt+↑/↓
- 레이어 삭제: ✕ 버튼 또는 Delete 키
- 속성 편집: 클릭 → 우측 패널
- 두께 모드: 상단 토글

### 4.4 RGB 자연 블록 스택
- 공통층은 같은 색상의 rect가 EML 위에 자연스럽게 올라앉는 형태
- 브릿지, 연결선 등 보조 장식 없음 — 색상만으로 연결 전달
- 클릭 시 동일 레이어 3개 rect 동시 선택 (통합 관리)
- 레이어 블록: 상단 엣지에 미묘한 하이라이트 그라디언트 허용 (내보내기 시 단색 폴백) (NFR-004 참조)

## 5. 기술 요구사항

### 5.1 기술 스택

| 항목 | 선택 | 근거 |
|------|------|------|
| **플랫폼** | Electron | 팀 역량(React/TS) 최대 활용, 가장 빠른 MVP, Windows 설치형 배포 |
| **프론트엔드** | React + TypeScript | 기존 팀 역량 |
| **렌더링** | SVG | 벡터 내보내기 완벽, React 친화적, 레이블 기본 지원 |
| **드래그앤드롭** | @dnd-kit/sortable | 유지보수 활발, 키보드 접근성 기본 지원 |
| **내보내기** | html-to-image (PNG), SVG 직렬화 | 표준 구현 |
| **상태 관리** | Zustand 또는 useReducer | Undo/Redo 히스토리 관리 |
| **데이터 저장** | JSON 파일 (네이티브 파일 다이얼로그) | Electron 파일 시스템 API |

### 5.2 데이터 모델

```typescript
interface Project {
  schemaVersion: string  // "1.0.0"
  metadata: { name: string; version: string; createdAt: string; updatedAt: string }
  viewMode: 'scientific' | 'presentation'
  structureMode: 'single' | 'rgb'
  palette: 'classic' | 'pastel' | 'vivid'  // 현재 적용 팔레트
  stacks: Stack[]
}

interface Stack {
  id: string
  label: string
  layers: Layer[]
}

interface Layer {
  id: string
  name: string
  role: 'cathode' | 'eil' | 'etl' | 'eml' | 'htl' | 'hil' | 'anode' | 'cgl' | 'custom'
  material: string
  thickness: number
  colorToken: string    // 팔레트 기반 토큰 (예: 'cathode', 'etl')
  customColor?: string  // 사용자 오버라이드 색상 (있으면 토큰보다 우선)
  groupId?: string
  appliesTo: ('r' | 'g' | 'b')[]
  channelOverrides?: {
    r?: { thickness?: number; material?: string; customColor?: string }
    g?: { thickness?: number; material?: string; customColor?: string }
    b?: { thickness?: number; material?: string; customColor?: string }
  }
}
```

### 팔레트 해석 규칙
1. layer.customColor가 존재하면 → customColor 사용 (팔레트 무시)
2. layer.customColor가 없으면 → project.palette + layer.colorToken으로 CSS Custom Property 참조
3. 팔레트 전환 시 customColor가 설정된 레이어는 색상 유지 (의도적 오버라이드 보존)
4. 내보내기 시 최종 resolved 색상값(computed color)을 인라인 스타일로 기록

### 5.3 제약사항
- Windows 전용 (macOS/Linux는 v2 이후 검토)
- 오프라인 동작 필수 (클라우드 의존 없음, LLM 기능 제외)
- Electron 번들 크기 ~150MB 허용

## 8. 리스크 및 가정

### 리스크

| 리스크 | 심각도 | 완화 전략 |
|--------|--------|----------|
| 실제 두께 모드: 극단적 비율 (1nm vs 200nm) | High | 최소 height 보장 + 로그 스케일 옵션 검토 |
| RGB 데이터 모델 복잡성 (tandem, top/bottom emission 등) | High | MVP는 FMM 단일 구조만, 변형은 v2 |
| SVG 내보내기 품질 (폰트, 스타일 인라인화) | Mid | 내보내기 전 인라인 스타일 변환 필수 |
| Windows 크로스 빌드 (Mac 개발 환경) | Mid | GitHub Actions CI 또는 Windows VM |
| 사용자가 "PPT로 충분" 판단 | High | 데이터 기반 편집 + 내보내기 품질로 차별화 |
| JSON 스키마 버전 관리 | Mid | schemaVersion 필드로 마이그레이션 지원 |
| 손상 파일 복구 | Mid | 로드 시 JSON 스키마 validation + 에러 시 사용자에게 경고 메시지. 자동백업은 v1.1 자동저장과 함께 도입 |
| Undo/Redo 메모리 증가 | Low | 히스토리 100개 제한 + 오래된 항목 자동 제거 |
| Windows DPI 스케일링 불일치 | Mid | devicePixelRatio 기반 렌더링 + QA에서 150%/200% 검증 |
| 두께 입력 검증 (0, 음수, 극단값) | Mid | 양수만 허용, 최대 10,000nm, 실시간 validation |

### 가정

| 가정 | 검증 방법 |
|------|----------|
| OLED 연구자가 전용 도구를 원한다 | 사용자 인터뷰 (v1 배포 후) |
| SVG 렌더링이 논문 품질을 충족한다 | QA에서 실제 논문 형식 검증 |
| Electron 번들 크기가 사용자에게 문제되지 않는다 | 베타 피드백 |
